Why it exists

An LLM only knows what it was trained on.

Ask a plain model about your company's return policy, last week's news, or a private PDF, and it has two bad options: guess (and hallucinate), or admit it doesn't know. Its knowledge is frozen at training time and has no access to your data.

✗ LLM alone

Answers from memory only. Confident even when wrong. Can't cite a source. Goes stale the moment facts change.

"What's our refund window?" → invents "30 days" 🤷

✓ LLM + RAG

Retrieves the real, relevant documents first and reads them while answering. Grounded, current, and able to point at the source.

"What's our refund window?" → reads policy.pdf → "14 days, per §4."

The pipeline

Retrieve, then generate — step through it.

A question doesn't go straight to the model. First it's turned into an embedding and used to search a vector database of your documents. The best-matching chunks are stapled onto the prompt. Then the LLM answers — using that context as its open book.

Build it once

Indexing vs. answering.

RAG has two phases. You index your documents once (offline), then answer queries against that index over and over (online).

Chunk & embed (offline). Split documents into passages, run each through an embedding model, store the vectors in a vector DB. docs → chunks → vectors
Embed the question (online). The user's query goes through the same embedding model so it lives in the same vector space. query → q-vector
Retrieve top-k. Find the chunks whose vectors are nearest to the query — typically by cosine similarity. search → top-k chunks
Augment the prompt. Paste those chunks into the prompt as context, alongside the original question. prompt = context + question
Generate. The LLM writes an answer grounded in the retrieved text — and can cite which chunk it used. LLM → grounded answer
